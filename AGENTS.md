# AGENTS.md

本仓库用于本地 `ai-ft-opc` 项目的自动化与线索收集实验。任何代理或自动化助手在本仓库工作时，必须优先遵守安全边界，避免影响真实服务、真实数据和真实潜在客户。

## 当前已知技术栈

- Docker Compose：用于本地编排服务。
- PostgreSQL：用于结构化业务数据、任务队列状态和后续线索数据。
- Redis：用于缓存、队列或轻量级状态协调。
- Qdrant：用于向量检索和相似度搜索。
- n8n：用于工作流编排和人工审核前后的自动化步骤。
- GitHub：用于代码托管、变更审查和协作。

## 工作原则

- 只修改用户明确要求修改的文件。
- 在修改前先理解现有结构，不假设未导出的 n8n 工作流或凭据已经在 Git 中完整存在。
- 文档、示例和流程说明必须清楚标注实验状态、人工审核点和安全限制。
- 不自动联系真实 leads，不自动发送邮件、私信、表单或任何外部触达。
- 不自动推送到远端仓库，除非用户明确批准。

## n8n 导出现状

真实 n8n 工作流和 credentials 目前尚未完整导出到 Git。仓库文档只能描述流程、边界和约定，不能替代 n8n 实例中的实际工作流与凭据备份。

## 明确禁止事项

- 绝不暴露 `configs/`、`.env`、n8n credentials、API key、token、cookie 或任何密钥。
- 绝不删除 Docker volumes。
- 绝不运行破坏性 SQL，例如 `DROP`、`TRUNCATE`、无条件 `DELETE`、无条件 `UPDATE`，或任何不可逆数据迁移。
- 绝不自动联系真实 leads。
- 绝不在没有用户明确批准的情况下 `git push`。
- 绝不修改 `docker-compose.yml`、数据库 SQL 文件、`.env` 文件、n8n 凭据或既有工作流逻辑，除非用户在当前任务中明确要求。

## 变更交付要求

- 优先提供小范围、可审查的变更。
- 涉及部署、迁移、工作流或真实数据的建议必须写明人工确认步骤。
- 涉及外部系统的操作必须默认保持只读或模拟模式。
- 如果发现实际 n8n 工作流或凭据未导出到 Git，应明确说明现状，不要伪造配置内容。
