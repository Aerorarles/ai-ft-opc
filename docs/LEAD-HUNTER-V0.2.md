# Lead Hunter V0.2

## 当前已知技术栈

- Docker Compose：本地服务编排。
- PostgreSQL：结构化数据与任务状态。
- Redis：缓存、队列或短期状态。
- Qdrant：向量索引与相似度检索。
- n8n：自动化工作流编排。
- GitHub：代码、文档和审查协作。

本文档定义 Lead Hunter V0.2 的安全流程草案。该流程用于从排队任务出发，收集候选公司，经过去重和人工审核后，再进入 lead intake 与评分阶段。

## 当前状态

- 当前仓库记录的是流程设计和安全约定。
- 真实 n8n 工作流和 credentials 尚未完整导出到 Git。
- 本文档不能替代 n8n 中的实际工作流备份。
- 所有真实外部触达必须由人工确认，系统不得自动联系真实 leads。

## 安全流程

```text
queued agent_tasks
  -> search source connector
  -> candidate company collection
  -> deduplication
  -> human review
  -> lead intake/scoring
```

## 阶段说明

### 1. queued agent_tasks

系统从排队的 `agent_tasks` 中读取待处理任务。任务应包含搜索目标、来源限制、执行状态和审计信息。

安全要求：

- 任务默认不触发真实外部触达。
- 任务执行前应记录来源、创建者和目标范围。
- 不应在任务内容中保存密钥、cookie 或 n8n credentials。

### 2. search source connector

搜索源连接器负责从允许的数据源中检索公司或市场信息。连接器可以是 n8n 节点、后端服务或人工导入数据。

安全要求：

- 优先使用公开、合规、低风险的数据源。
- 不绕过网站访问限制。
- 不使用未授权账号或泄露的凭据。
- 不把搜索源 credentials 提交到 Git。

### 3. candidate company collection

候选公司集合保存原始发现结果和基础字段，例如公司名、网址、行业、地区、来源 URL 和采集时间。

安全要求：

- 原始数据与后续评分数据应可追踪来源。
- 不把个人敏感信息作为默认采集目标。
- 不把候选公司直接视为可联系 lead。

### 4. deduplication

去重阶段基于公司名、域名、地区、来源 URL、相似文本或向量匹配合并重复候选。

安全要求：

- 去重结果应保留来源记录，避免丢失审计信息。
- 自动合并应保守，低置信度结果进入人工审核。
- 不因相似度结果自动覆盖人工标注。

### 5. human review

人工审核是进入 lead intake 前的必要关口。审核人确认候选公司是否相关、是否合规、是否允许进入后续评分。

安全要求：

- 未通过人工审核的候选不得进入真实触达流程。
- 审核界面应明确显示来源、采集时间和风险提示。
- 人工审核通过不等于允许自动联系。

### 6. lead intake/scoring

通过审核的候选进入 lead intake 与评分。评分可以考虑行业匹配度、公司规模、地理位置、公开信号、相似客户特征和业务优先级。

安全要求：

- 评分仅用于排序和内部判断。
- 评分结果不得自动触发真实邮件、电话、表单提交或社交平台私信。
- 任何真实触达必须经过单独人工批准。

## 数据存储建议

- PostgreSQL：保存 `agent_tasks`、候选公司、去重结果、审核状态、lead intake 和评分记录。
- Redis：保存短期任务状态、锁或缓存。
- Qdrant：保存候选公司描述、网页摘要或相似度检索向量。
- n8n：编排任务读取、连接器调用、人工审核通知和后续入库。

## 明确禁止事项

- 绝不暴露 `configs/`、`.env`、n8n credentials 或任何密钥。
- 绝不删除 Docker volumes。
- 绝不运行破坏性 SQL。
- 绝不自动联系真实 leads。
- 绝不在没有明确批准的情况下 push 到 GitHub。

## V0.2 完成标准

- 可以从 `agent_tasks` 中安全读取排队任务。
- 可以通过搜索源连接器收集候选公司。
- 可以保存候选公司集合并保留来源。
- 可以执行保守去重。
- 可以进入人工审核。
- 只有人工审核通过后，才进入 lead intake 与 scoring。
- 不包含自动真实触达能力。
