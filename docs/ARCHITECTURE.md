# 架构说明

本文档描述 `ai-ft-opc` 当前已知的本地系统架构。当前仓库以 Docker Compose 为本地运行入口，结合 PostgreSQL、Redis、Qdrant、n8n 和 GitHub 形成自动化实验环境。

## 当前已知组件

- Docker Compose：负责本地服务编排和依赖启动。
- PostgreSQL：保存结构化数据，例如任务、候选公司、线索 intake、评分结果和审核状态。
- Redis：用于缓存、短期队列、执行状态或工作流协调。
- Qdrant：保存向量数据，用于相似公司、文本语义检索或 lead 相关匹配。
- n8n：编排搜索、去重、审核、入库和评分等自动化流程。
- GitHub：保存代码、文档和可导出的配置，作为协作与审查入口。

## 高层结构

```text
GitHub 仓库
  -> Docker Compose 本地服务
    -> PostgreSQL 结构化数据
    -> Redis 缓存与队列状态
    -> Qdrant 向量索引
    -> n8n 工作流编排
```

## 数据与控制流

1. 用户或系统创建待处理任务，例如 `agent_tasks` 中的排队任务。
2. n8n 或后端代理读取任务，并调用搜索源连接器。
3. 搜索结果进入候选公司集合。
4. 系统执行去重和基础清洗。
5. 人工审核确认候选是否可进入 lead 流程。
6. 通过审核的候选进入 lead intake 与评分环节。

## n8n 导出现状

真实 n8n 工作流和 credentials 目前尚未完整导出到 Git。仓库中的文档只能描述目标流程、边界和约定，不能视为生产工作流的完整备份。

## 安全边界

- 绝不暴露 `configs/`、`.env`、n8n credentials 或任何密钥。
- 绝不删除 Docker volumes。
- 绝不运行破坏性 SQL。
- 绝不自动联系真实 leads。
- 绝不在没有明确批准的情况下 push 到 GitHub。

## 设计假设

- 本地环境优先用于开发、验证和人工审核前的实验。
- 所有真实外部触达必须由人工确认后执行。
- 数据迁移和部署必须可审查、可回滚，并避免默认修改真实数据。
