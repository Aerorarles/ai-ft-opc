# 数据迁移说明

## 当前已知技术栈

- Docker Compose：本地服务编排。
- PostgreSQL：结构化数据与任务状态。
- Redis：缓存、队列或短期状态。
- Qdrant：向量索引与相似度检索。
- n8n：自动化工作流编排。
- GitHub：代码、文档和审查协作。

本文档记录 `ai-ft-opc` 中数据库迁移的安全约定。当前已知数据库组件为 PostgreSQL，Redis 和 Qdrant 也可能保存运行状态或索引数据。

## 迁移目标

数据库迁移应满足：

- 变更可审查。
- 执行前可预估影响范围。
- 执行后可验证。
- 出现问题时有回滚方案。

## PostgreSQL 迁移原则

- 优先使用显式迁移文件管理表结构、索引和约束。
- 迁移前确认连接目标，避免误连生产或共享数据库。
- 对大表变更应评估锁表、耗时和回滚影响。
- 对数据修复脚本应先写只读查询，确认影响行数后再执行。

## 严禁操作

- 绝不运行破坏性 SQL。
- 绝不运行未经审查的破坏性 SQL。
- 绝不运行无条件 `DELETE` 或无条件 `UPDATE`。
- 绝不直接执行 `DROP`、`TRUNCATE` 或不可逆 schema 变更。
- 绝不删除 Docker volumes 来“重置”数据库。
- 绝不暴露 `configs/`、`.env` 或数据库凭据，也绝不把这些内容提交到 Git。
- 绝不自动联系真实 leads。
- 绝不在没有明确批准的情况下 push 到 GitHub。

## Redis 与 Qdrant 注意事项

- Redis 可能保存队列、缓存和短期状态，清空前必须确认影响。
- Qdrant 可能保存向量集合和索引，删除 collection 前必须有备份或可重建方案。
- 不应把清空缓存、删除索引或重建 collection 当作默认迁移步骤。

## 推荐迁移流程

1. 编写迁移说明，明确目的、影响表、影响字段和回滚方式。
2. 在只读模式下查询当前状态和影响范围。
3. 在本地或测试环境执行迁移。
4. 验证数据完整性、索引状态和应用读写路径。
5. 经人工批准后，才允许在共享环境执行。

## n8n 与迁移关系

真实 n8n 工作流和 credentials 尚未完整导出到 Git。涉及 n8n 的数据迁移或 workflow 状态变更时，必须先人工导出备份，并确认 credentials 不会被泄露。
